# How Memory Works / mem 快速理解

## 这份文档是干什么的

如果你第一次接触 `codex-mem`，这份文档是给你快速建立直觉的。

适合谁看：

- 日常使用 `codex-mem` 的用户
- 想快速理解 mem 原理的开发者
- 需要知道“什么时候查记忆、什么时候看代码”的协作者

适合什么时候看：

- 第一次接触这个项目
- 不理解 `scope`、相关项目、AGENTS 行为时
- 想知道普通用户最推荐的使用方式时

不适合拿来做什么：

- Go 架构设计
- CI / release 检查
- MCP 传输协议调试

它不讲底层协议细节。
它只回答几个最常见的问题：

- mem 到底是干什么的
- 平时怎么用
- `scope` 到底是什么意思
- 什么时候需要关心作用范围
- 哪些命令是给维护者用的，哪些是日常使用入口

## 一句话理解

`codex-mem` 是给 Codex 用的“长期记忆层”。

它主要做三件事：

1. 在你开始新一轮工作时，帮你恢复这个仓库之前的重要上下文
2. 在你工作过程中，保存长期有效的结论、经验和约束
3. 在你暂停或结束时，保存交接记录，方便下次继续

## 日常怎么用

对大多数用户来说，只要记住这四种场景：

1. 开始前：恢复这个仓库之前的上下文
2. 有长期有效结论时：保存成长期记忆
3. 结束前：保存交接记录
4. 想查以前内容时：搜索记忆或查看最近记录

你通常不需要记工具名。
直接和 Codex 说人话就行。

例如：

```text
先恢复这个仓库之前的上下文，再继续当前任务。
```

```text
把这个结论保存成长期记录，后面还会用到。
```

```text
结束前保存一条交接记录，写清楚下一步做什么。
```

```text
搜索一下以前和远程 MCP 部署相关的历史记忆。
```

## mem 里存的是什么

最核心的是两类内容：

- `MemoryNote`
  适合保存长期有效的信息，比如设计决策、bug 根因、修复经验、约束、偏好、待办
- `Handoff`
  适合保存“这次做到哪了，下一步是什么”的交接记录

可以把它粗略理解成：

- note 更像“长期知识”
- handoff 更像“接班说明”

## `scope` 到底是什么意思

`scope` 这个词在规范里很多，但对普通用户可以直接翻成：

**这条记忆属于哪个范围**

最常见的是三层：

- `system`
  一整个系统或产品线
- `project`
  一个项目
- `workspace`
  你当前这份本地仓库/工作目录

但你平时完全没必要老想着这三个词。
更实用的理解是：

- 默认先看当前仓库
- 需要时再扩到当前项目
- 只有明确允许时才跨项目查

## 我需要手动指定范围吗

大多数情况下，不需要。

正常流程是：

1. 你先说“恢复这个仓库之前的上下文”或者“开始一个新的带记忆会话”
2. 系统自动识别你当前在哪个仓库/项目里
3. 之后保存 note 或 handoff 时，就默认保存到这个范围下面

只有这些情况，才值得你明确提“范围”：

- 当前目录可能识别不准
- 你有多个 clone / worktree
- 你想先确认系统识别到了哪个仓库/项目
- 你明确要查相关项目，而不只是当前仓库

## 如果我在 A 项目里开发，但想参考 B 项目，应该怎么做

这是一个典型的“跨项目参考”场景。

最实用的原则是：

- 如果这次任务一开始就明确依赖 B 项目，可以在开场时一起加载
- 如果只是开发到一半才想参考 B 项目，按需搜索就行

### 适合开场就一起加载的情况

例如：

- A 项目要接 B 项目的接口
- A 项目依赖 B 项目的配置、协议、生成代码或部署方式
- 这次任务从一开始就离不开 B 项目上下文

这时可以直接这样说：

```text
先恢复当前 A 仓库之前的上下文。这次任务依赖 B 项目，也一起加载和 B 项目相关的历史记忆，并标明哪些内容来自 B 项目。
```

### 适合中途按需搜索的情况

例如：

- 一开始主要还是做 A
- 只是开发到一半时，想看看 B 以前是怎么处理类似问题的

这时更适合这样说：

```text
我现在在 A 项目里开发，这次任务和 B 项目有关。请搜索一下 B 项目里与当前任务相关的历史笔记或交接记录，并明确标出哪些结果来自 B 项目。
```

### 这里有个很重要的限制

跨项目检索不是默认全开的。

正常情况下系统会优先：

1. 当前仓库
2. 当前项目
3. 明确允许的相关项目

所以你需要在提示词里明确表达：

- 这次任务和 B 项目有关
- 允许一起加载或搜索相关项目记忆
- 希望结果标明来源项目

### 一句话建议

可以直接记成：

- 开场强依赖 B：启动时一起加载
- 中途偶尔参考 B：需要时再搜索

## 如果 A 和 B 以前从没建立过关联，第一次该怎么做

这是当前实现里最容易让人困惑的地方。

### 先说结论

如果 A 项目和 B 项目以前没有任何关联记忆，A 不能只靠一句“我想参考 B”就自动稳定识别并接入 B。

当前实现更接近下面这个逻辑：

1. 先让系统识别出 B 的 canonical `project_id`
2. 再在 A 项目里保存一条 note，明确记录 “A 和 B 有关系”
3. 这条 note 里要带上 B 的 `related_project_ids`
4. 从这以后，A 才能更稳定地把 B 当成相关项目一起检索

### B 不是按目录名引用的

这里要特别注意：

- 不是本地目录名
- 不是随手写的 “B项目”
- 不是任意仓库别名

按当前实现，真正稳定的引用方式是：

- B 的 canonical `project_id`

原因很简单：

- 本地目录名可能会改
- 一个项目可能有多个 clone / worktree
- 规范里也不把本地目录名当成可靠的项目身份

### 第一次手工建立 A -> B 关联的建议步骤

#### 第一步：先去 B 仓库，让系统识别 B

在 B 仓库里先做一次：

- 恢复上下文
- 或开始一个新的带记忆会话

目的不是马上查记忆，而是让系统先识别 B 的 canonical `project_id`。

#### 第二步：回到 A 仓库开始会话

回到 A 仓库后，正常开始当前任务。

#### 第三步：在 A 里保存一条“关联说明” note

这条 note 可以写得很直白，例如：

```text
A 项目的支付流程依赖 B 项目的订单接口和字段约定。后续凡是涉及支付提交流程、订单状态映射或接口字段兼容性的问题，都需要允许一起参考 B 项目的历史记忆。
```

这条 note 的关键不只是正文，而是底层要把 B 的 canonical `project_id` 记进：

- `related_project_ids`

### 建立之后会发生什么

一旦这条关系被存下来，后面在 A 项目里就可以更自然地说：

```text
我现在在 A 项目里开发，这次任务和 B 项目有关。请搜索一下 B 项目里与当前任务相关的历史笔记或交接记录，并明确标出哪些结果来自 B 项目。
```

或者：

```text
先恢复当前 A 仓库之前的上下文。这次任务依赖 B 项目，也一起加载和 B 项目相关的历史记忆，并标明哪些内容来自 B 项目。
```

### 当前这套机制的局限

这部分目前更偏工程实现，还不算特别“傻瓜式”。

对普通用户来说，心里想的是：

- A 依赖 B

但系统底层现在真正存的是：

- `related_project_ids = ["proj_xxx"]`

所以从体验上说，当前版本还缺一层更友好的“按仓库自然建立关联”的封装。

## 我想了解 B 项目，到底是在看 B 的记忆，还是在看 B 的代码

这是两个不同的问题。

### 看 B 的记忆

这指的是去看 B 以前沉淀下来的：

- note
- handoff
- 设计决策
- bug 根因
- 修复经验
- 任务交接记录

如果你想知道的是：

- B 以前为什么这么做
- B 之前踩过什么坑
- B 有没有长期约束或历史决策

那你应该优先查：

- B 的 memory

### 看 B 的代码

这指的是直接去看 B 仓库里的：

- 源码
- 配置
- 接口定义
- schema
- 部署脚本

如果你想知道的是：

- B 现在代码到底怎么写
- 某个接口现在的真实实现是什么
- 某个字段、配置或逻辑目前长什么样

那你应该直接：

- 看 B 的代码

### 最实用的判断方法

可以直接这么区分：

- 想知道“以前为什么这样做” -> 查记忆
- 想知道“现在代码到底是什么” -> 看代码

### 如果 B 还没有任何记忆

那就没有可加载的 B 项目记忆。

这时候：

- 你仍然可以直接看 B 的代码
- 也可以从现在开始在 B 项目里逐步积累 memory

所以 `codex-mem` 不是代码阅读的替代品。
它更像是“帮助你少重复踩坑、少重复理解背景”的长期记忆层。

## B 的 `project_id` 怎么拿

最实用的方式不是猜，也不是看目录名。

而是：

- 直接到 B 仓库里，让系统识别一次当前仓库范围

因为 `memory_resolve_scope` 或 `memory_bootstrap_session` 的返回结果里，本来就会带 `scope`，而 `scope` 里包含 canonical `project_id`。

### 推荐做法

先进入 B 仓库，然后对 Codex 说：

```text
先确认一下当前这个 B 仓库的范围，并把识别到的 project_id 告诉我。
```

或者：

```text
先恢复当前这个 B 仓库之前的上下文，并把当前 scope 里的 project_id 告诉我。
```

拿到这个 `project_id` 之后，再回到 A 仓库建立 A -> B 的关联，会比直接猜一个名字稳得多。

## 什么时候该保存 memory

适合保存：

- 长期有效的结论
- 可复用的 bug 修复经验
- 以后还会反复碰到的约束
- 接下来要继续做的任务状态

不适合保存：

- 临时聊天内容
- 一闪而过的想法
- 原始 transcript
- 明确不该持久化的私密内容

## 用户到底会接触哪些功能

日常最常用的是这些：

- `memory_bootstrap_session`
  恢复上下文并开始新会话
- `memory_save_note`
  保存长期记忆
- `memory_save_handoff`
  保存交接记录
- `memory_search`
  搜索历史记忆
- `memory_get_recent`
  看最近的笔记和交接记录

相对不需要普通用户经常关心的是这些：

- `memory_resolve_scope`
  更多是“先确认系统识别到哪个仓库/项目”
- `memory_get_note`
  按 id 读取单条完整记录
- `memory_install_agents`
  初次接入时做一次即可
- `serve` / `serve-http` / `doctor`
  这是部署和维护命令，不是普通用户的日常工作入口

## 如果项目里已经有 `AGENTS.md`，会发生什么

这取决于安装时使用的模式。

### `safe`

这是默认最稳妥的模式。

- 如果 `AGENTS.md` 不存在，就创建
- 如果已经存在，就跳过，不覆盖
- 会返回一个 warning，告诉你已有文件被保留了

### `append`

这个模式适合“保留原文件，再追加一段由 mem 管理的说明”。

- 如果 `AGENTS.md` 不存在，就创建
- 如果已经存在，就在后面追加一段受管理的 block
- 如果这个 block 之前已经追加过了，就不会重复追加

### `overwrite`

这个模式会直接用模板内容覆盖现有文件。

- 适合你明确知道自己要重写 `AGENTS.md`
- 不是默认行为
- 普通情况下不要随手用

### 最实用的建议

如果你不确定，就用 `safe`。

可以把它理解成：

- `safe` = 有就保留，没有就创建
- `append` = 尽量保留原文件，再补一段
- `overwrite` = 整份重写

## 一套最省心的使用方式

你可以把它固定成这个习惯：

1. 开始工作前，先恢复当前仓库上下文
2. 做出重要结论后，顺手保存长期记忆
3. 暂停前，保存一条交接记录
4. 下次回来，先恢复再继续

这样就够了。

## 如果只想记最短版本

只记下面四句就行：

- `先恢复这个仓库之前的上下文`
- `把这个保存成长期记录`
- `结束前保存一条交接记录`
- `搜索一下以前和 XXX 相关的记忆`

## FAQ: `memory_save_note` 和 `memory_save_handoff` 有什么区别

这两个都属于“保存记忆”，但用途不一样。

### `memory_save_note`

更适合保存会在后面反复复用的内容，例如：

- 设计决策
- bug 根因和修复经验
- 以后还会继续约束实现的规则
- 值得长期保留的发现

可以把它理解成：

- “知识卡片”
- “长期结论”

如果你想表达的是“这个结论以后还会有用”，通常更适合存成 note。

### `memory_save_handoff`

更适合保存这一次任务的续做信息，例如：

- 这次已经做了什么
- 当前做到哪里
- 下一步该做什么
- 还有哪些风险、问题、未完成项

可以把它理解成：

- “交接记录”
- “续做检查点”

如果你想表达的是“下次回来怎么接着做”，通常更适合存成 handoff。

### 最实用的判断方法

可以直接这样区分：

- 想保存“以后很多次都可能会用到的经验或结论” -> `memory_save_note`
- 想保存“这次任务下次如何继续” -> `memory_save_handoff`

很多时候两者可以一起保存：

1. 先把值得长期保留的结论存成 note
2. 再把当前任务进度和 next steps 存成 handoff

## FAQ: `workspace` 和 `project` 有什么区别

可以先记成一句话：

- `workspace` = 你当前这份本地工作目录 / clone / worktree
- `project` = 这些 workspace 所属的同一个项目身份

### `workspace`

它更接近：

- 这台机器上当前这个路径下的仓库副本
- 你现在正在工作的这份本地工作区

如果同一个项目有多个本地副本，例如：

- 多个 clone
- 多个 worktree
- 不同磁盘路径下的同一仓库副本

那么它们通常属于不同的 workspace。

### `project`

它更接近：

- 一个更稳定的项目身份
- 不依赖某一个具体本地路径的项目级上下文

同一个项目的不同 clone 或 worktree，通常都归到同一个 project。

### 最实用的理解方式

可以这样记：

- `workspace` 更像“这份本地工作副本的当前进度”
- `project` 更像“这个项目整体的长期记忆”

### 为什么要分这两层

因为系统恢复上下文时，通常会优先：

1. 看当前 workspace 的 handoff 和 notes
2. 如果不够，再回退到同一个 project 下的内容

这样做的好处是：

- 先优先恢复你这份本地工作区最直接相关的上下文
- 不够时，再补项目级的长期经验和历史记录
